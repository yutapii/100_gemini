システムは以下の構造であることを前提とする。

```text
[プロジェクトルート]/
├── 000_concept/          # 設計・マニュアル
├── 010_ai/               # AI設定
├── 030_config/           # 共通設定 (JSON/YAML)
├── 040_security/         # 機密情報 (ID/Pass/Token) ※.gitignore必須

│   ├── config/           # 個別設定 (設定分離)
│   ├── output/           # 出力データ
│   └── [script].py       # 実行ロジック (単一責任)
└── 999_trash/            # 廃止・バックアップ (検査対象外)
```

## 2. 規律検査基準 (Discipline)

以下の「物理的制約」を1バイトの妥協なく遵守せよ。

| 項目 | 制限値 | 判定基準 | 備考 |
| :--- | :--- | :--- | :--- |
| **行数制限** | **500行以内** | `wc -l` | 単なる「行数合わせ」は厳禁。500行に近づいた時点で**「機能分解」と「モジュール分割」**を設計レベルで検討・実行せよ。 |
| **一行文字数** | **80文字以内** | `awk 'length($0) > 80'` | 可読性とサイドバイサイド比較の容易さを担保する。 |
| **ファイル命名** | **小文字+蛇腹** | `[a-z0-9_]+` | OS間互換性と検索性を重視。 |

## 3. セキュリティ検査基準 (Security)

「漏洩は死」を合言葉に、以下の項目を徹底点検せよ。

- **.gitignore**: `040_security/` が除外されているか。
- **ハードコード排除**: コード内に以下のキーワードが含まれていないか。
  - `password`, `secret`, `api_key`, `token`, `auth`, `credentials`
  - ※これらは必ず `040_security/` または環境変数から読み込むこと。
- **絶対パス排除**: `/Users/saitoyutaka/` などの個人環境に依存したパスが含まれていないか。
  - `Path(__file__)` や `os.path.expanduser('~')` を使用せよ。

## 4. メンテナンス性検査基準 (Maintainability)

100年使い続けるための「設計の品格」を問う。

- **設定の分離**: ロジック（.py/.js）と設定（.json/.yaml）が分離されているか。
  - 判定：`config/` フォルダが存在し、変数がそこから読み込まれている。
- **localStorage使用禁止 (No LocalStorage)**: ブラウザの `localStorage` へのデータ保存は**厳禁**とする。
  - 理由：デバッグを困難にし、サーバー側とのデータ不整合（SPOF）を引き起こすため。
  - 対策：永続化が必要な場合は、必ずサーバー側のAPIを経由し、JSONファイル等へ書き込むこと。
- **冪等性 (Idempotency)**: 何回実行しても、同じ入力に対して同じ結果（または安全なスキップ）が得られるか。
- **証跡の出力**: 処理結果が `output/` または `.log` に記録され、事後検証が可能か。

## 5. 検査ツール (自動化スクリプト)

以下のコマンドで一括検査を実行せよ。

```bash
# プロジェクトルートで実行
./100_inspection_tools/inspect_v3_iron_gate.sh [対象ディレクトリ]
```

### 判定の見方
- **✅ 合格 (100点)**: 全ての規律を遵守。リリース許可。
- **⚠️ 注意**: セキュリティキーワード検出（目視確認が必要）。
- **❌ 不合格**: 行数・文字数超過。即時リファクタリング。
